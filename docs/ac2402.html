AC24.02Cyber

Mettre en œuvre les outils fondamentaux de sécurisation d’une infrastructure du réseau

Ce que j’ai fait

Dans le cadre de la SAE 4.Cyber.01, j'ai configuré et appliqué des listes de contrôle d'accès (ACL) étendues et restrictives sur notre commutateur cœur de réseau de niveau 3 (SR1) afin de cloisonner de manière étanche les zones utilisateurs de moindre confiance par rapport à nos zones d'administration sensibles.

Pourquoi je l’ai fait

Le routage inter-VLAN permet par défaut à n'importe quel segment réseau de communiquer avec un autre. Pour sécuriser l'infrastructure, il est crucial d'appliquer le principe du moindre privilège au niveau réseau. Les postes du réseau de Production ou de Service ne doivent jamais pouvoir initier un flux vers les serveurs et postes de gestion de l'Administration, évitant ainsi la propagation latérale d'une éventuelle compromission.

Comment je l’ai fait

J'ai implémenté l'ACL nommée BLOCK_SERVICE_PROD. Pour ne pas bloquer le routage dynamique, le trafic ospf est autorisé. Les réseaux d'administration (192.168.10.0/24 et 192.168.11.0/24) disposent d'un accès complet. Pour les autres zones, j'ai mis en place un mécanisme pseudo-stateful : l'accès n'est autorisé que si le bit TCP established est actif (ce qui indique une réponse à une connexion déjà initiée par l'administrateur) ou s'il s'agit d'un message ICMP echo-reply. Toute tentative d'initiation de connexion vers la zone d'administration est explicitement rejetée par la règle finale deny ip any any.

Mes difficultés

La principale difficulté résidait dans le fait qu'un filtrage trop strict bloquait également les réponses légitimes des serveurs vers les outils d'administration (par exemple, un ping de diagnostic restait sans réponse). L'analyse de l'ordre séquentiel des lignes d'ACL et l'utilisation rigoureuse des types de paquets ICMP (distinction entre echo et echo-reply) m'ont permis de résoudre ce problème de connectivité bidirectionnelle sécurisée.

Ce que j’en ai appris

J'ai maîtrisé l'écriture d'ACL étendues complexes sous l'environnement Cisco IOS, compris la logique de filtrage par liste blanche et assimilé l'utilisation des indicateurs d'état TCP (established) pour sécuriser l'infrastructure réseau de manière granulaire.

Ce que je ferais autrement

Les ACLs classiques sur commutateur analysent les paquets ligne par ligne sans véritable table d'état dynamique en mémoire. Pour une infrastructure de production d'entreprise, je préférerais déléguer ce filtrage de zone à un pare-feu applicatif (Next-Gen Firewall) ou activer des fonctionnalités de pare-feu basées sur des zones (Zone-Based Firewall - ZBF) sur le routeur afin de bénéficier d'une inspection stateful complète et native.

Trace ciblée : Configuration de la liste d'accès étendue sur le commutateur L3 SR1 
ip access-list extended BLOCK_SERVICE_PROD 
 permit ospf any any ! OSPF non bloqué 
 permit ip 192.168.11.0 0.0.0.255 any ! Admin droit : accès total 
 permit ip 192.168.10.0 0.0.0.255 any ! Admin gauche : accès total 
 permit tcp any 192.168.10.0 0.0.0.255 established ! Réponses TCP vers Admin gauche 
 permit icmp any 192.168.10.0 0.0.0.255 echo-reply ! Réponses ping vers Admin gauche 
 permit icmp any 192.168.11.0 0.0.0.255 echo-reply ! Réponses ping vers Admin droit 
 permit tcp any 192.168.11.0 0.0.0.255 established ! Réponses TCP vers Admin droit 
 deny ip any any ! Tout le reste bloqué
